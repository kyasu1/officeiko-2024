## Project Overview

This is the website for **Office IKO (オフィスイコー)**, a Japanese pawnshop and foreign exchange service. It offers pawn services, Western Union remittance, and precious metal market data.

Built with **elm-pages 3.x** (a full-stack Elm SSR framework), deployed to **Netlify**.

## Commands

```bash
npm start        # Start development server (elm-pages dev)
npm run build    # Production build (elm-pages build)
```

No test suite exists. For code quality:
```bash
npx elm-format src/ app/ --yes   # Format Elm code
npx elm-review                   # Lint/static analysis
```

Netlify build command (set in netlify.toml):
```
export ELM_HOME="$NETLIFY_BUILD_BASE/cache/elm" && npm install && npm run build
```

## Architecture

### Framework: elm-pages

elm-pages is a full-stack Elm framework with file-based routing. Pages are SSR-capable with `BackendTask` for server-side data fetching at build time or request time.

**Source directories** (defined in `elm.json`):
- `app/` — Route modules and framework entry points
- `src/` — Shared business logic, layouts, and utilities
- `.elm-pages/` — Auto-generated by the framework (do not edit)

### Routing (`app/Route/`)

Each file corresponds to a URL route. Dynamic segments use `Slug_.elm` naming (e.g., `Post/Slug_.elm` → `/post/:slug`). Routes implement a standard pattern:

```elm
-- Required exports for each route:
type alias RouteParams = { ... }
type alias Data = { ... }
type alias ActionData = { ... }

route : StaticSemanticRoute ... RouteParams Data ActionData
data : BackendTask FatalError Data
head : StaticPayload Data ActionData RouteParams -> List Head.Tag
view : App Data ActionData RouteParams -> Shared.Model -> View (PagesMsg Msg)
```

### Shared State (`app/Shared.elm`)

Global state shared across all pages: mobile menu toggle, current date, and precious metal market prices. The `Shared.Model` is passed to every route's `view` function.

### Data Files

- `market.json` — Precious metal prices (gold/platinum/silver); updated by automated commits
- `ticket.json` — Pawn ticket data
- `content/posts/` — Markdown blog posts with YAML frontmatter
- `content/selling.json`, `content/access.json`, `content/special-holiday.json` — Static content

### Layouts (`src/Layout/`)

Layout modules are pure view functions that separate rendering logic from route data-fetching logic. Routes fetch data then pass it to the corresponding layout module for rendering.

### Effects (`app/Effect.elm`)

Custom effect system wrapping `Cmd` for side effects. Used for form submissions and fetchers — keeps route modules decoupled from Cmd/Sub infrastructure.

### Styling

TailwindCSS v4 (via `@tailwindcss/vite` plugin) combined with `rtfeldman/elm-css`. Inline Tailwind class strings appear throughout Elm view code. Global styles in `style.css`.

### Serverless Functions (`functions/`)

Netlify Functions handle SSR. `functions/render/index.mjs` and `functions/server-render/index.mjs` are the server-side rendering endpoints — these are managed by elm-pages and the Netlify adapter.

### Code Generation (`script/`, `codegen/`)

`script/src/AddRoute.elm` and `AddStaticRoute.elm` are CLI scripts using `elm-codegen` to scaffold new route modules. The `codegen/Gen/` directory contains auto-generated type-safe wrappers (currently deleted from git, regenerated on build).

### Configuration

- `elm-pages.config.mjs` — Vite config, Netlify adapter, Google Tag Manager (GTM-55GV2Z), head tag template
- `index.ts` — Client entry point; passes GTM dataLayer and current date as flags to Elm
- `.env` — Contains `VITE_NEWT_TOKEN` and `VITE_STRAPI_TOKEN` (not committed)
